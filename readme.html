// Quickstart

In 5 Minuten live

Railway-Deployment via Git-Push. Keine Docker-Konfiguration nötig.

bash · Installation 
# 1. Repository klonen
git clone https://github.com/dein-user/telllmeeedrei.git
cd telllmeeedrei

# 2. Dependencies installieren
pip install -r requirements.txt

# 3. Umgebungsvariablen setzen (siehe .env.example)
cp .env.example .env
# → öffne .env und trage alle Keys ein

# 4. Lokal starten
uvicorn main:app --reload --port 10000

# 5. Auf Railway deployen
railway up
python · Bot in eigenes Projekt einbinden 
# Einzelne Module importieren und wiederverwenden
from brain import save_text, load_entry, list_entries
from bot_ai import generate_response, transcribe_voice
from imgrem import generate_image
from search import web_search, format_search_results_for_user
from guard  import can_process_text

# Beispiel: KI-Antwort generieren
async def my_handler(chat_id, user_message):
    decision = can_process_text(chat_id, user_message)
    if not decision.allowed:
        return decision.message

    reply = await generate_response(chat_id, user_message)
    return reply

# Beispiel: Bild generieren
async def make_image(prompt):
    img_buffer = await generate_image(prompt, width=1024, height=1024)
    if img_buffer:
        return img_buffer  # BytesIO-Objekt, direkt sendbar

# Beispiel: Brain-Eintrag speichern
result = await save_text(chat_id="12345", text="Notiz", title="Meeting")
print(result)  # "Text gespeichert. ID: ..."
python · Eigenen Command registrieren 
# In main.py – nach dem Muster der vorhandenen Handler:
from telegram.ext import CommandHandler
from mein_modul import mein_handler

application.add_handler(CommandHandler("meincommand", mein_handler))

# Minimal-Handler Template:
async def mein_handler(update, context):
    chat_id = str(update.effective_chat.id)
    payload = " ".join(context.args).strip()
    if not payload:
        await update.message.reply_text("Nutzung: /meincommand <text>")
        return
    loading = await update.message.reply_text("⏳ Verarbeite...")
    result = await meine_logik(payload)
    await context.bot.edit_message_text(
        chat_id=chat_id,
        message_id=loading.message_id,
        text=result
    )

// Code-Audit

Stärken, Schwächen & Empfehlungen

Detaillierter Review aller 30+ Module. Grün = gut gelöst, Gelb = verbesserungswürdig, Rot = kritisch.

✅

Polling-Architektur mit Watchdog

main.py — Exponential Backoff, 5-Minuten-Watchdog der Webhooks automatisch löscht, starke Task-Referenz verhindert GC-Abbrüche. Sehr robust für Railway-Deployments.

✓ SOLID
✅

Modell-Fallback-Kaskade

bot_ai.py — LLaMA-4-Scout → LLaMA-3.3-70B → Mixtral → Gemma2. Saubere Fehlerbehandlung mit 503/404-Erkennung. Kein Single-Point-of-Failure.

✓ SOLID
✅

Guard / Rate-Limiter

guard.py — Action-spezifische Token-Bucket-Implementierung, automatische Bucket-Bereinigung, Privacy-Mode, Text-Moderation. Vollständig ohne externe Dependencies.

✓ SOLID
✅

Brain + VectorBrain

brain.py + vectorbrain.py — Supabase-Integration mit fehlertoleranter Fallback-Logik. Hybride Suche (Vektor 82% + Lexikalisch 18%). Sentence-Transformer mit lokalem Hash-Fallback.

✓ SOLID
✅

Universeller Parser / Konverter

parser.py + dv.py — Saubere Trennung von Extraktion und Konvertierung. Unterstützt 12 Zielformate. Code-Fallback via codeinterpreter.py bei Parser-Fehlern. Sehr vollständig.

✓ SOLID
✅

Agent Tool-Loop

agent.py — Sauberer generischer Agentenloop mit Tool-Schema-Generierung, async Tool-Ausführung und Step-Limit. Korrekte Message-History-Verwaltung für Tool-Calls.

✓ SOLID
⚠️

Doppelte Dateien: claude_code.py vs. claude_code_fixed.py

Beide Dateien sind fast identisch. Nur claude_code.py wird in main.py importiert. claude_code_fixed.py und handle_claude_code.py (alte Version mit kaputtem from olko import) sollten gelöscht werden.

⚠ CLEANUP
⚠️

Doppelte Dateien: brainlist_handler.py vs. brainlist_handler_fixed.py

Inhaltlich identisch. Nur eine davon wird importiert. Die _fixed.py-Variante kann gelöscht werden, der Import in main.py ist bereits auf die nicht-fixed-Version gesetzt.

⚠ CLEANUP
⚠️

Hardcoded API-Key in suno_music.py

SUNO_API_KEY = "1e321224..." ist direkt im Code. Sollte auf os.getenv("SUNO_API_KEY") umgestellt werden, bevor das Repository öffentlich gemacht wird.

⚠ SICHERHEIT
⚠️

Hardcoded HF_TOKEN in free_music.py

HF_TOKEN = "hf_GmKchic..." direkt im Code statt os.getenv. Zwar auch via os.getenv gelesen, aber der Fallback-String steht im Klartext im Repository.

⚠ SICHERHEIT
⚠️

handle_claude_code.py importiert nicht-existente Module

from olko import claude_code_handler und last_generated_code/create_download_buffer fehlen als Import. Diese Datei ist die alte Version und wird nicht verwendet – sollte gelöscht werden.

⚠ TOTES CODE
⚠️

musika_music.py: Nicht produktionsreif

Erwartet geklontes Musika-Repository unter /app/musika. Der Download läuft synchron im Konstruktor und blockiert den Start. Auf Railway wird das bei jedem Deploy über 2 GB laden. Deaktivieren oder lazy-load.

⚠ PERFORMANCE
⚠️

Timeout-Bug in ttv26.py

asyncio.get_event_loop().time() - asyncio.get_event_loop().time() ergibt immer 0 und bricht nie ab. Die MAX_POLL_TIME_SECONDS-Prüfung ist damit wirkungslos. Muss mit start_time = asyncio.get_event_loop().time() vor der Schleife gefixt werden.

⚠ BUG
⚠️

youtube.py parallel zu ytscript.py

Zwei YouTube-Module mit überlappender Funktionalität. youtube.py verwendet die YouTube Data API v3, ytscript.py die youtube-transcript-api. In main.py wird nur ytscript.py verwendet. youtube.py ist toter Code.

⚠ CLEANUP
⚠️

voice_mini_app.py: Hardcodierte Railway-URL

wss://telllmeeedrei-production.up.railway.app/livevoice/ws/voice steht fest im JS. Wird automatisch gebrochen wenn der Railway-Projektname geändert wird. Besser via Template-Variable oder dynamisch aus window.location lesen.

⚠ MAINTENANCE
🔴

claude_code.py: Anthropic-Client ist nicht async

client.messages.create() der Anthropic-Library ist synchron. Der await-Aufruf in claude_code.py funktioniert nur weil die Library einen Coroutine-Wrapper hat – aber es blockiert den Event-Loop. Lösung: await asyncio.to_thread(client.messages.create, ...) oder AsyncAnthropic verwenden.

✗ KRITISCH
🔴

brain.py: Supabase-Client ist synchron, wird in async aufgerufen

Alle get_supabase()-Aufrufe laufen direkt im Event-Loop ohne asyncio.to_thread. Bei vielen gleichzeitigen Requests blockiert das den Polling-Loop. Alle Supabase-Aufrufe sollten in asyncio.to_thread() gewickelt werden.

✗ KRITISCH
🔴

Kein Logging der nicht verwendeten voice_mini_app-Import-Fehler

main.py importiert from voice_mini_app import app as voice_mini_app ganz oben – noch vor FastAPI-Init. Falls voice_mini_app.py einen Fehler hat, crasht der gesamte Bot beim Start ohne klare Fehlermeldung. Lazy-Import oder try/except schützt davor.

✗ KRITISCH

// Datenbank

Supabase Schema

Zwei Tabellen: brain_entries für alle Inhalte, brain_vectors für semantische Suche.

sql · Supabase Setup 
-- Tabelle für alle Brain-Einträge
CREATE TABLE brain_entries (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     text NOT NULL,
  entry_type  text NOT NULL,   -- 'chat' | 'safe_text' | 'file'
  title       text,
  content     text,
  metadata    jsonb,
  created_at  timestamptz DEFAULT now()
);

-- Index für schnelle User-Lookups
CREATE INDEX brain_entries_user_idx ON brain_entries(user_id);

-- RLS aktivieren
ALTER TABLE brain_entries ENABLE ROW LEVEL SECURITY;
CREATE POLICY allow_all ON brain_entries FOR ALL USING (true);

-- Vektoren-Tabelle für semantische Suche
CREATE TABLE brain_vectors (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     text NOT NULL,
  entry_id    text NOT NULL,
  title       text,
  content     text,
  metadata    jsonb,
  embedding   vector(384),  -- pgvector Extension nötig
  UNIQUE(user_id, entry_id)
);

-- pgvector Extension aktivieren (im Supabase-Dashboard)
CREATE EXTENSION IF NOT EXISTS vector;
